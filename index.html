                    # 特点

                    Lad拥有数十种功能，并且高度可配置.

                    # 微服务

                    这些微服务已针对安全、性能和优雅的重载资源进行预配置.

                    点击链接查看示例

                    • Webapp server → web.js (opens new window)
                    • API server → api.js (opens new window)
                    • Job scheduler → bree.js (opens new window)
                    • Proxy server → proxy.js (opens new window)

                    # 前端

                    • eslint-plugin-compat (opens new window) 和browserslist (opens new window)用于处理浏览器兼容性(有关默认配置，请参阅.browserslistrc (opens new window))
                    • Pug (opens new window) 模板引擎 (由于你可以很方便的使用Moon (opens new window), Vue (opens new window), React (opens new window), 或 Angular (opens new window),通常你不需要它)
                    • Gulp (opens new window) (latest version 4.x)
                    • Sass (opens new window)
                    • PostCSS (opens new window) (with font-magician (opens new window), import-url (opens new window), font-grabber (opens new window), base64 (opens new window), and cssnext (opens new window) pre-configured)
                    • Bootstrap (opens new window)
                    • Font Awesome (opens new window)
                    • SpinKit (opens new window)
                    • SweetAlert2 (opens new window)
                    • Dense (opens new window)
                    • Waypoints (opens new window)
                    • LiveReload (opens new window)
                    • …

                    # 后端

                    • Redis, sessions, 和 flash toast 以及 模态 SweetAlert2 (opens new window) messages (使用 ioredis (opens new window) 它支持 Cluster (opens new window), Sentinel (opens new window)等)
                    • 基于Koa的webapp 和 API servers (生产环境使用HTTP/2)
                    • 内置分页 (使用 ctx-paginate (opens new window))
                    • RESTful API 使用 BasicAuth 和 versioning
                    • 自动化的计划任务和便于人阅读的语法(由 Mongoose (opens new window) 和 Bree (opens new window) 提供支持)
                    • Passport-based 的身份校验 和 group-based (Unix-like) 权限许可
                    • 受Stripe启发的错误处理
                    • 有通用的数据库插件的 Mongoose 和 MongoDB
                    • 邮件模板引擎和本地渲染 Nodemailer (opens new window)
                    • 内置代理，不再需要Nginx 反向代理或 Apache 虚拟主机
                    • 内置的国际化翻译支持多种语言 (详细配置)
                    • 使用Google Translate进行自动短语翻译
                    • 可用于简单SEO的站点地图生成器
                    • …

                    # 翻译

                    框架的国际化最终由自动翻译完成.

                    • 翻译内建常量，不需重复翻译.
                    • 翻译Webapp 的错误消息和模板.
                    • 翻译Emails
                    • 翻译API 响应
                    • 翻译数据库错误消息
                    • 翻译身份校验错误
                    • …

                    # Email 引擎

                    我们的Email引擎使用了邮件模板 email-templates (opens new window) (这也是Lad作者的作品)!

                    • 本地测试测试你的emails,同时能快速的浏览器快速渲染.
                    • 自动创建内联css,提供跨浏览器和扩平台的email客户端支持.
                    • 在email模板设计中使用 Bootstrap (opens new window) .
                    • 重用已经存在的CSS和Webapp样式
                    • 可以任意选择模板引擎 (默认是Pug)
                    • 使用代码在email中渲染自定义的字体 (opens new window)
                    • 使用代码添加Font-awesome图标 (opens new window)
                    • 自动禁用email客户端缓存 (opens new window)
                    • 任何图片都能添加并正确渲染.
                    • 无需笨拙的嵌入式图像 CID附件
                    • …

                      # 开发

                      命令行输入 npm start (or yarn start) 这可以列出所有你能运行的脚本命令.

                      该命令接收一个参数<task>, 后面再跟 all会生成、监听、重新编译上面提到的所有微服务microservices.

                      打开 http://localhost:3000 (opens new window) 即可测试查看效果!

                        npm start all

                        yarn start all

                        # 生产环境

                        强烈推荐生产环境下使用 SemaphoreCI (opens new window), PM2 (opens new window), 和 Digital Ocean (opens new window).

                        1. 我们提供了预配置文件ecosystem.json (opens new window) deployment file (opens new window). 你只需要按需修改你服务器的IP, hostname, 以及其它的信息.

                        2. 确保你的项目构建时是在 NODE_ENV=production 环境, 比如: NODE_ENV=production npm run build (或者yarn build);这会每个 koa-manifest-rev (opens new window)对应创建一个 build/rev-manifest.json.

                        3. 本地测试可以通过 npm (opens new window) 或 yarn (opens new window)全局安装PM2 (opens new window), 然后运行以下命令:

                          NODE_ENV=production pm2 start

                        4. 参阅下面 持续继承和代码覆盖率测试 和 教程，以获得相关教程.

                        5. 如果你需要将程序发布到Amazon S3/Cloudfront，请设置环境变量AWS_CF_DOMAIN 并且设置 NODE_ENV=production. 请运行npm start publish-assets (或 yarn start publish-assets). 该命令会自动设置NODE_ENV=production 通过 cross-env.

                        # Provisioning

                        请参阅 ansible (opens new window)文件夹了解我们的Ansible (opens new window) 配置和行动指南(playbooks), 这里面有如何配置服务器的详细信息。

                        在使用 Ansible (opens new window) 行动指南时，我们建议你安装yamllint (opens new window) 并在编辑器中对其进行配置.

                        另外， 当你计划修改行动指南时ansible-lint (opens new window) 是一个很有用的lint检查工具. 注意到当前的行动指南(playbooks)有几处lint错误.

                        首先，你需要准备好 Ubuntu 18.04 LTS 64-bit 的服务器(群), 可以使用 Digital Ocean (opens new window), Linode (opens new window), Vultr (opens new window),或者其它的云服务器. 这些新服务器你需要添加 SSH 访问秘钥.

                        接下来，参阅部署了解如何配置和部署服务器.

                        # 部署

                        1. 配置主机host, 可以拷贝 hosts.yml文件模板:

                          cp ansible/playbooks/templates/hosts.yml hosts.yml

                        1. 编辑该配置文件，配置你的服务器别名和IP地址. 如做负载均衡你可以每组添加不止一台服务器. 关于我们推荐的服务器别名命名方法，请参阅命名 Naming Convention 文档. 该文件自动被git忽略, 如果你的私有库想提交该文件, 请从.gitignore中手动移除 hosts.yml.

                          vim hosts.yml

                        2. 环境配置，可以拷贝 env 模板文件:

                          cp ansible/playbooks/templates/env .env.production

                        1. 编辑该配置文件，可用的环境变量配置请参阅官方Lad (opens new window) 文档 (或参阅 .env.defaults (opens new window)). 你需要在你的编辑器上打开该文件 并给任何标记了TODO的字段设置值. 记得保留双引号.

                          vim .env.production

                        2. 使用自动模板生成器来生成 pm2 (opens new window) ecosystem files (opens new window). 创建 ansible-playbook.js (opens new window) 来加载 .env.production 环境变量，这些环境变量通过 @ladjs/env (opens new window) 加载进process.env中， 随后在行动指南(playbooks)中要用到. 这是一种优先级高、简单的也是在Ansible中唯一的dotenv方法. 新的ecosystem-api.json, ecosystem-bree.json, ecosystem-web.json 文件会创建在你的根目录下. 如果再添加或更改 IP 地址，只需重新运行此命令即可.

                          node ansible-playbook ansible/playbooks/ecosystem.yml -l 'localhost'

                        3. 设置web或IP服务器(群)(详情参阅 patterns and ansible-playbook flags docs (opens new window)). 如果你完整(或部分)的运行行动指南(playbooks)或这下边的其它东西, 随之第二次运行可能出现失败, 这是因为我们通过安全强化来阻止 root 用户访问. 要解决此问题, 可重新运行该命令但不附加 -e 'ansible_user=root', 此情况默认user为创建者.

                          node ansible-playbook ansible/playbooks/http.yml -e 'ansible_user=root' -l 'http'

                        4. 设置 Bree 服务器(群):

                          node ansible-playbook ansible/playbooks/bree.yml -e 'ansible_user=root' -l 'bree'

                        5. 设置 Redis 服务器:

                          node ansible-playbook ansible/playbooks/redis.yml -e 'ansible_user=root' -l 'redis'

                        6. 设置 MongoDB 服务器:

                          node ansible-playbook ansible/playbooks/mongo.yml -e 'ansible_user=root' -l 'mongo'

                        7. 给所有服务器设置 GitHub deployment keys. 注意deployment-keys目录被git忽略了, 所以如果私有库需要上传 deployment-keys需手动从.gitignore文件中移除.

                          node ansible-playbook ansible/playbooks/deployment-keys.yml -l 'http:bree'

                        8. 打开github中库的"Settings"页, 点击 "Deploy keys", 然后复制秘钥到 deployment-keys 目录，从而为每台服务器添加一个部署秘钥deployment key. 如果是macOS上, 可使用 pbcopy命令复制每个文件的内容到剪贴板. 按tab键快速填充，或者替换为自己的服务器名称和路径.

                          cat deployment-keys/api-1-li-dal.forwardemail.net.pub | pbcopy

#
# NOTE: repeat the above command for all servers
# and after running the command, it will copy
# the key to your clipboard for you to paste as
# a new deploy key (make sure to use read-only access)
#

                        9. 为所有服务器设置 PM2 部署目录:

                          pm2 deploy ecosystem-web.json production setup

                          pm2 deploy ecosystem-api.json production setup

                          pm2 deploy ecosystem-bree.json production setup

                        10. 在Namecheap (opens new window)上创建一个 SSL 证书 (我们建议是 5年的通配型证书), 设置证书, 然后在电脑上下载解压证书(通过email发给你的). 我们不建议使用像 LetsEncrypt (opens new window) 和 certbot等工具， 因为当你使用(或扩展到)服务器集群及负载均衡时会很复杂. 换句话讲，我们已经尝试过 lsyncd 结合 crontab 来进行certbot证书的续期和自动检查的方法. 另外, 使用该方法服务器会面临停机，因此80 和 443 端口需要关闭， 以便生成certbot证书. 这不是一种可靠的方法，并且只需每年续订一次证书要简单得多，并且还使得使用负载均衡器变得微不足道。 更好的办法是： 使用Namecheap (opens new window) 得到一个便宜的 SSL 证书, 然后运行如下所示的少量命令. 此命令将提示你输入下载证书的绝对路径. 一年后重新生成证书? 简单重复这一步骤就行, 注意证书文件上别设置密码. 当使用openssl 命令 (详情参见Namecheap (opens new window))时, 如果你注册通配符证书,你需要使用*.example.com 句点后边跟星号

                          ansible-playbook ansible/playbooks/certificates.yml

                          Important: 如果将来续订或更改证书，则在运行上一个命令后，随后需要按如下方式重新加载进程:

                          #
# NOTE: See the "Important" note above BEFORE running this command.
#       This command ONLY APPLIES for certificate renewals/changes.
#
pm2 deploy ecosystem-web.json production exec "pm2 reload web"
pm2 deploy ecosystem-api.json production exec "pm2 reload api"

                        11. (可选) 如果你需要支持自动翻译，只需创建Google应用证书文件并本地存储. 下面的命令会提示你输入绝对路径 (e.g. /path/to/client-profile.json). 关于普通话的详细信息，请参阅mandarin (opens new window).

                          ansible-playbook ansible/playbooks/gapp-creds.yml -l 'http:bree'

                        12. 复制 .env.production 并在AWS服务器上创建配置文件:

                          node ansible-playbook ansible/playbooks/env.yml -l 'http:bree'

                        13. 在所有服务器上运行和初始化部署:

                          pm2 deploy ecosystem-web.json production

                          pm2 deploy ecosystem-api.json production

                          pm2 deploy ecosystem-bree.json production

                        14. 保存进程列表，以便服务器重启时能自动重新启动这些进程:

                          pm2 deploy ecosystem-web.json production exec "pm2 save"

                          pm2 deploy ecosystem-api.json production exec "pm2 save"

                          pm2 deploy ecosystem-bree.json production exec "pm2 save"

                        15. 在浏览器上访问来测试你的 Web 和 API 服务器 (点击 "proceed to unsafe" 忽略证书警告certificate warning).

                        16. 为 Web 和 API 服务器主机名以及相应的 IP 地址配置 DNS 记录.

                        17. 在浏览器上访问来测试你的 Web 和 API 服务器(打开隐私窗口InPrivate). 这儿应该没有证书警告 (类似于步骤15).

                        18. (可选)出于安全目的，移除本地 .env.production文件. 如果这样做，请确保有备份, 或者在将来销毁服务器之前安全地从服务器备份.

                          rm .env.production

                        19. (可选) 删除本地下载的步骤 11中指定的本地证书文件. 如果这样做，请确保有备份, 或者在将来销毁服务器之前安全地从服务器备份.

                        20. 完成. 如果你需要再次部署, 修改后push到github master 分支然后再根据步骤14操作. 我们建议你阅读Ansible上手指南Ansible getting started guide (opens new window), 这能帮助你更深入了解运行在所有或指定服务器上的命令如 ansible all -a "echo hello"

                          # 环境变量

                          通过 dotenv (opens new window) 的名为 @ladjs/env (opens new window)的配置包, 按照 Twelve-Factor (opens new window)原则，使你的Lad项目配置十分容易.

                          我们使用下列3个包来管理配置:

                          • dotenv-extended (opens new window) - 允许我们在名为.env.schema的文件中创建 .env 定义 (也叫做"schema")
                          • mustache (opens new window) - 允许我们在.env 和 .env.defaults 配置文件中使用 Mustache templating language (opens new window)
                          • dotenv-parse-variables (opens new window) - 从process.env中自动解析变量类型 (例如: FOO=4 会设置成 process.env.FOO = 4 FOO设置成Number类型而不是 String)

                          配置管理的优先级顺序如下:

                          1. config/index.js 文件中的内容 (从process.env中读取环境变量)
                          2. config/environments/ 目录下的文件内容 (为每个环境变量设置默认值, e.g. 你可以设置 NODE_ENV=staging 然后会加载config/environments/staging.js)
                          3. 用来覆盖默认值或者设置所需的环境变量 (e.g. NODE_ENV=production)
                          4. .env 中设置的环境变量
                          5. .env.defaults中的环境变量

                          顺序为环境配置文件，环境变量， .env 文件.

                          基本上， dotenv (opens new window)如果检测到它作为环境变量传递，则不会设置环境变量。(译者注:它指代什么不清楚,这句话怪怪的)

                          config (opens new window) 文件夹下及默认值.env.defaults (opens new window)的内容如下.

                          • NODE_ENV - (options: development, production default: development) - node 运行的环境
                          • PROXY_PORT - (default: 8080) - 代理请求用的代理端口 (see ladjs/proxy (opens new window))
                          • HTTP_PROTOCOL - (defaults: http recommend: https) - http 协议(protocol)
                          • HTTP_PORT - (defaults: 80 recommend: 443) - http port used for http requests
                          • WEB_PROTOCOL - (default: http) - ladjs/web (opens new window) application protocol
                          • WEB_HOST - (default: localhost) - ladjs/web (opens new window) application host
                          • WEB_PORT - (default: 3000) - ladjs/web (opens new window) application port
                          • WEB_URL - (default: ://:) - web application absolute URI
                          • WEB_SSL_KEY_PATH - ladjs/web (opens new window) file path to your SSL key file
                          • WEB_SSL_CERT_PATH - ladjs/web (opens new window) file path to your SSL certificate file
                          • WEB_SSL_CA_PATH - ladjs/web (opens new window) file path to your SSL certificate authority file
                          • API_HOST - (default: localhost) - ladjs/api (opens new window) host
                          • API_PORT - (default: 4000) - ladjs/api (opens new window) port
                          • API_PROTOCOL - (default: http recommend: https) - ladjs/api (opens new window) protocol

                          # SSL 配置

                          要为 Web 或 API 服务器配置 SSL，只需在 .env 文件中设置它们或将它们作为环境变量传递

                          Web 服务器:

                          • WEB_PROTOCOL - 必须设置为 https
                          • WEB_SSL_KEY_PATH - SSL 秘钥的文件路径 (e.g. /home/deploy/.ssl/web-key.pem)
                          • WEB_SSL_CERT_PATH - SSL 证书的文件路径 (e.g. /home/deploy/.ssl/web-cert.pem)
                          • WEB_SSL_CA_PATH (可选) - SSL 证书颁发机构文件的文件路径 (e.g. /home/deploy/.ssl/web-ca-cert.pem)

                          API 服务器:

                          • API_PROTOCOL - 必须设置为 https
                          • API_SSL_KEY_PATH - SSL 秘钥的文件路径 (e.g. /home/deploy/.ssl/api-key.pem)
                          • API_SSL_CERT_PATH - SSL 证书的文件路径 (e.g. /home/deploy/.ssl/api-cert.pem)
                          • API_SSL_CA_PATH (可选) - SSL 证书颁发机构文件的文件路径 (e.g. /home/deploy/.ssl/api-ca-cert.pem)

                          # 持续继承和代码覆盖率测试

                          我们强烈推荐使用 SemaphoreCI (opens new window) 作为持续集成, [Codecov][https://codecov.io/gh] 作为代码覆盖率测试的工具.

                          Here are the simple steps required to setup SemaphoreCI (opens new window) with [Codecov][https://codecov.io/gh]:

                          1. 访问 SemaphoreCI (opens new window) 并注册一个免费账号.

                          2. 如果你的项目在Github上, 从Github导入项目到 SemaphoreCI

                          3. 使用如下构建设置，来配置你在SemaphoreCI上的项目.

                            如果你使用yarn, 把npm 换为 yarn

                            • Language: JavaScript
                            • Node.js version: 10+ (latest LTS)

                              Note you can also add to Setup the script nvm install latest to install latest version if SemaphoreCI does not provide it from the drop-down 请注意，如果 SemaphoreCI 的下拉列表中未提供，您可以将 nvm install latest 添加到 Setup 以安装最新版本

                            • Setup: npm install
                            • Job 1: npm run test-coverage
                            • After job: npm run coverage

                          4. 访问 [Codecov][https://codecov.io/gh] 注册一个免费账号.

                          5. 添加你的项目到 Codecov 并复制 Token 到剪贴板.

                          6. 到 SemaphoreCI 上你项目的设置 Settings, 添加环境变量 CODECOV_TOKEN (内容就是你上边剪贴板里的)

                          7. 在 SemaphoreCI 上运行测试构建（重建最新版本"Rebuild last revision"）并检查以确保您在 Codecov 上的代码覆盖率报告正确上传

                          8. 确保您的 README.md 文件具有正确呈现的构建状态和代码覆盖率标记（如果您的 GitHub 存储库是私有的，您将需要使用来自每个提供商的不同标记链接）

                          # Contributing

                          Interesting in contributing to this project or testing early releases?

                          1. Follow all of the above Requirements

                          2. You will need to fork and clone this repository locally

                          3. After forking, follow these steps:

                            cd lad
yarn install
cd template
yarn install
yarn start

                          If you'd like to preview changes to the README.md file, you can use docute.

                          yarn global add docute-cli
cd lad
docute ./

                          Then visit http://localhost:8080 (opens new window) in your browser.
